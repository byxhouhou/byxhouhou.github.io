一个五年嵌入式工程师在 AI 时代的思考
发表于2026-05-28|个人思考
最近使用了很火的 Codex，它很大程度上颠覆了我对编程工作的认知。 原来我可能需要 1-2 个月才能完成的编码工作，现在它不到半天就可以做完，而且完成质量甚至比我自己写得更稳定、更完整。 过去公司很看重的“代码产出能力”，正在发生变化。现在，只要你知道基本架构，理解模块之间的关系，AI 就可以帮你生成大量代码。这个时代对于程序员而言，真正重要的能力不再只是把每一行代码手写出来，而是理解整体架构、理解上下游数据交互、理解具体业务逻辑，并且能够判断 AI 生成的东西是否正确。 这让我有一种被解放的感觉。 我终于不用再为了一个循环边界、一个入参长度、一个接口细节花掉半天时间。我有了更多精力去学习原来感兴趣但一直没有时间深入的领域。AI 也可以帮助我更快地梳理知识体系。我很喜欢把自己的理解讲给 AI 听，再让它来纠正我，这种学习方式非常高效。 从嵌入式开发到更大的创作边界我是一个嵌入式开发工程师。按过去的职业路径，我平时很少会接触网页、UI、美术、视频、游戏这些领域。 但是在 Codex 的帮助下，我已经可以自己完成小程序开发、网页制作、简单的游戏 demo 等等。当然，token 也确实挺贵的（笑）。 过去拦在我面前的东西，很多并不是创造力本身，而是不同编程语言的差异、开发环境的搭建、陌生知识带来的畏惧感。现在这些门槛正在被 AI 消解。 我可以更自由地实现自己的创意和想法，而不是只局限在单片机这一个细分领域里。这对我来说，是一种很强烈的扩展感。 更进一步，我还可以把自己的内容输入给 AI，让它帮我生成视频、图片、音乐。也就是说，我不再只能是一个程序员。我也可以是视频制作者、图像制作者、音乐制作者，甚至是一个更综合的创作者。 在 AI 时代，只要你愿意投入注意力，很多原本离你很远的能力都会变得可以触达。 知识在 AI 时代会越来越接近免费，真正宝贵的是注意力。你要选择自己想专注什么领域，也要决定自己愿意把时间投入到哪里。我现在每天高强度使用 AI，最大的感受反而是：时间不够用。 AI 改变了我的学习方式回到 GPT 的使用，最近这几个月是我个人认知成长最快的一段时间。 每天高强度提问，让我开始接触社会学、经济学、心理学等多个领域。这些原来都是作为理工科背景的我比较欠缺的部分。现在每天都能学到不同领域的新知识，再把它们融入自己的知识结构里，这件事本身就很有快乐。 我越来越觉得，所谓文理之争，本质上是社会根据知识分工形成的分类方式。知识本身是用来描述世界、解释规律、理解现象的，不应该被简单粗暴地划分成“文科”或“理科”。 我真的很建议不同学科背景的人，去主动涉猎一些陌生领域。这件事非常有趣。 难道你不好奇自己是怎么被社会观念影响的吗？ 难道你不好奇计算机是怎么用无数个 0 和 1 构建出这个数字世界的吗？ 难道你不好奇为什么炒菜时放一点鸡精，味道就会明显变鲜吗？ 这个世界里值得理解的东西太多了，而 AI 是一个非常好的助手。 以后，所有公开的通用性知识都会越来越没有获取门槛。你和各个知识领域的大拿之间，可能只差一次和 AI 的深入对话。 程序员真正需要提高的能力对我来说，AI 并不是简单替代程序员的工具。它更像是在重新定义程序员的能力重心。 过去更重要的是：你能不能把代码写出来。 现在更重要的是： 你是否理解问题本身； 你是否知道系统该如何分层； 你是否清楚上下游如何交互； 你是否能判断 AI 生成的代码是否可靠； 你是否能把业务逻辑讲清楚； 你是否具备持续学习新领域的能力。 AI 会让普通编码变得更便宜，但也会让架构判断、业务理解、系统设计、问题定义这些能力变得更重要。 对于嵌入式工程师来说也是一样。我们不能只停留在寄存器、驱动、接口调用这些局部细节里。底层能力仍然重要，但更重要的是理解系统：硬件、驱动、OS、通信协议、业务逻辑、诊断、云端、工具链之间到底是如何协作的。 AI 能帮我们写代码，但它不能替我们决定真正要解决什么问题。 这可能就是我作为一个五年嵌入式工程师，在 AI 时代最直接的感受。

UDS在AUTOSAR与FreeRTOS项目中的工程架构与开发流程整理
发表于2026-05-24|Automotive
前言本文不重点介绍 UDS 协议定义和功能介绍，相关内容可以AI询问或者参考文档。 ISO14229 可直接参考： https://www.iso.org/standard/72439.html 本文更关注： UDS 在真实车载项目中的位置、上下游接口、工作流与开发问题。 主要结合： AUTOSAR CP架构下UDS功能实现 FreeRTOS系统中UDS功能手写代码实现 目前实现UDS也就是这两种方式，那让我们开始 1.UDS在整体软件架构中的位置先建立整体认知。 很多新人会把 UDS 理解成： “几个 SID 服务” 实际上在工程里，它是一个横跨通信、状态机、权限、故障、刷写、存储的中间层模块。 1.1AutoSar架构介绍autoSar CP整体的软件架构如下： 它将软件分了如下几层 灰色：应用与硬件边界Application（应用层）:实际业务逻辑,通常对应SWC（Software Component）.UDS的很多DID的读写逻辑，DTC的检测逻辑都会在这里交给各个业务实现。 Microcontroller（微控制器硬件）：这就是指物理世界中的芯片了，比如STM32、NXPS32K等 橙色：RTE（软件连接层）RTE（Runtime Environment）：RTE 是 AUTOSAR 的中间连接层，连接了SWC和BSW，屏蔽了模块间直接调用。SWC不需要考虑BSW的具体实现，只需要调用对应的RTE接口就可以。 紫色：Service Layer（服务层）Service Layer（服务层）：提供系统级基础服务，比如ECU启动、模式管理、看门狗管理都在这里实现，UDS的DCM和DEM功能也在这里实现 绿色：ECU Abstraction Layer（ECU抽象层）ECU Abstraction Layer（ECUAL）：ECU抽象层，位于驱动层和服务层之间，这一层屏蔽具体的硬件差异，CanIf、EthIf等都在这一层实现 红色：MCAL（硬件驱动层）MCAL（Microcontroller Abstraction Layer）：这是最接近硬件的部分，直接操作MCU寄存器，像Mcu、Port、Dio、Can、Spi、Fls、Adc都会在这里去配置实现 1.2 UDS在Autosar中的架构在对AutoSarCP架构有一个基本了解后，让我们看下UDS在这个架构中如何工作的，这是自己画的诊断数据流向的图 这里举几个例子有助于你更好的理解CAN 诊断读取 VIN例如维修人员使用 CANoe、诊断仪或 OEM Tester 读取车辆 VIN。 诊断仪发送 22 F190 命令后，数据首先通过 CAN/CANFD 总线 传输到车机 ECU。ECU 内部的 CAN 控制器接收到物理报文，由 CAN 通信栈完成报文接收、多帧组包和路由。DCM 将请求解析为“读取 VIN”，随后调用应用层获取 VIN 数据（通常来自 RAM、NvM 或 Flash 持久化存储）。最后 ECU 将生成的响应报文重新通过 CAN 总线发送回诊断仪。 清除故障码（DTC） 维修工具执行“清除故障码”操作时，诊断仪会向 ECU 发送 UDS 清除命令。命令经过 CAN 总线 到达车机，DCM 识别该请求属于故障管理服务后，会通知 DEM 清除当前故障状态、冻结帧或扩展故障信息。如果故障信息需要掉电保存，DEM 会进一步调用 NvM 将更新后的数据写入 Flash 或 EEPROM。完成处理后，ECU 再通过 CAN 网络返回清除结果。 1.3 AutoSar中的DCM和DEM在整个 AUTOSAR Classic Platform（AUTOSAR CP） 中，UDS 诊断功能主要由 DCM（Diagnostic Communication Manager） 与 DEM（Diagnostic Event Manager） 两个 BSW 模块承担实现。 简单来说： DCM 负责外部诊断通信与诊断服务处理； DEM 负责 ECU 内部故障事件与 DTC 管理。 下面简单介绍一下这两个核心 BSW 模块。 DCM（Diagnostic Communication Manager）DCM 是 AUTOSAR 中的 诊断通信管理模块，负责实现大部分 ISO14229 UDS 服务逻辑。 当维修人员使用 诊断仪、CANoe、OEM Tester 等工具，通过 CAN / CANFD / DoIP 向车机 ECU 发送诊断命令时，请求经过通信栈后最终进入 DCM 进行处理。 DCM 主要负责： 诊断会话管理（0x10 Session Control） 安全访问（0x27 Security Access） DID 数据读写（0x22 / 0x2E） Routine Control（0x31） NRC 负响应生成 刷写流程控制（0x34 / 0x36 / 0x37） 与 Application / SWC 的数据交互 例如，维修人员通过诊断仪发送： 122 F190 读取 VIN 请求。 数据首先通过 CAN/CANFD 总线 传输到车机 ECU，经过 Can Driver → CanIf → CanTp → PduR 后到达 DCM。 DCM 识别出： 12SID = 0x22ReadDataByIdentifier 随后通过 RTE 调用 Application DID Callback 获取 VIN 数据（通常来自 RAM、NvM 或 Flash 持久化存储），最后生成响应报文并返回给诊断仪。 DEM（Diagnostic Event Manager）DEM 是 AUTOSAR 中的 故障诊断管理模块，负责 ECU 内部故障事件与 DTC 生命周期管理。 与 DCM 更偏向“外部诊断通信”不同，DEM 更关注： ECU 运行过程中产生的内部故障状态管理。 例如，车辆正常运行过程中，即使没有连接诊断仪，车机 ECU 也可能检测到： CAN 通信超时 Sensor 数据异常 电压过高/过低 Flash 读写失败 网络通信故障 此时，业务模块或底层监控逻辑会向 DEM 上报错误事件。 DEM 负责维护： Event 状态 DTC 生命周期 Pending / Confirmed DTC Freeze Frame Extended Data Aging / Debounce 必要时，DEM 还会通过 NvM 将故障信息保存到 Flash 中，实现掉电持久化保存。 例如，车辆运行过程中检测到 CAN 通信异常。 业务模块调用： 1Dem_ReportErrorStatus() DEM 更新故障状态后，可进一步调用： 1NvM → MemIf → Fee/Ea → Flash Driver 最终将故障记录写入 MCU Flash。
